# API Factory — Premium MCP Specification

**Version 1.0** · what makes every server in the API Factory catalog "premium" instead of slop.

Every server in our catalog follows this spec. The Creator and Polisher agents both read this file when generating and reviewing code. The Premium Checklist gates publish.

## The 8 Premium Differentiators

### 1. Consistent error envelope
Every error returned from a tool MUST use the `err()` helper from `lib/premium.js`.

```js
return err(ErrorCodes.INVALID_INPUT, 'limit must be 1-100', {
  hint: 'Try limit:50',
  retryable: true,
  docs: 'https://...'
});
```

The host LLM gets BOTH a readable line and a structured `_meta.error` payload, so it can self-correct without re-prompting.

**Codes** (use these — don't invent new ones unless truly needed):
`invalid_input`, `not_found`, `rate_limited`, `upstream_error`, `internal`, `permission`, `timeout`, `unsupported`, `conflict`, `payment_required`.

### 2. Dual-output responses
Every successful response uses `ok(payload, meta?)`. This produces:
- `content[0].text` — human/LLM-readable rendering
- `_meta` — structured payload for downstream tool chaining

Never hand-build `{ content: [...] }` arrays.

### 3. Built-in introspection — `about`, `health`, `examples`
Auto-registered by `registerIntrospection(server, {...})`. Do NOT remove these tools.
- `about` — name, version, useCases, signature
- `health` — uptime, ready state, runtime info
- `examples` — runnable example args per tool

### 4. Rich tool descriptions
Every tool's `description` field is multi-line and contains:
- 1-line summary
- `When to use:` — heuristic for the LLM
- `Args:` — what each arg does (short)
- `Returns:` — what comes back

Why: Claude/Cursor display this verbatim. It IS the SDK docs for the LLM.

### 5. Pagination contract
Any tool returning a list/collection MUST use `paged(items, { cursor, limit })`. Always include `cursor` (string|null) and `limit` (1-500, default 50) in the input schema.

### 6. Progress notifications for long ops
Use `progress(server, token, current, total, message)` for any op > 1s. Hosts that don't support progress get a silent no-op.

### 7. Dry-run mode for destructive/expensive tools
Add `dry_run: z.boolean().optional()` to the schema. Use the `dryRun()` guard at the top of the handler.

### 8. `safe()`-wrapped handlers
Every domain handler MUST be wrapped with `safe()`. This converts any uncaught throw into a clean `internal` error envelope instead of crashing the JSON-RPC channel.


## What the Polisher checks

The `polisher` agent runs after the Tweaker has fixed contract failures. It:

1. Lints `server.js` against this spec (programmatic checks in `premiumChecklist.js`)
2. Auto-rewrites weak tool descriptions
3. Auto-populates the `__examples` map if missing
4. Re-asserts the introspection tools are present
5. Rewrites the README using our voice

If any **critical** check fails after polishing, the publish gate refuses.

## Why this matters

MCP is a young ecosystem. Most servers on Smithery today are 1-day hackathon code with raw stack traces, no docs, and 1 tool. **Servers that follow this spec feel like SDKs**, not toys. That's our edge.
